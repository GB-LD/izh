# Story 1.5a: Schemas Zod et types

Status: ready-for-dev

## Story

As a user,
I want my data to be validated and protected against corruption,
so that the app never crashes or loses data due to malformed localStorage content.

## Acceptance Criteria

1. **AC1 — TaskSchema** : `src/schemas/task.ts` exporte un `TaskSchema` Zod avec les 13 champs suivants :
   - `id` : UUID v4 string
   - `title` : string, min 1 caractère
   - `status` : enum `inbox | backlog | active | archived`
   - `quadrant` : enum `q1 | q2 | q3 | q4`, nullable
   - `createdAt` : datetime ISO 8601 string
   - `classifiedAt` : datetime ISO 8601 string, nullable
   - `completedAt` : datetime ISO 8601 string, nullable
   - `flowDurationMs` : number, nullable
   - `sourceFlux` : enum `flux1 | flux2 | flux3 | flux4 | manual`, nullable
   - `classificationMethod` : enum `assisted | manual`, nullable
   - `userOverride` : boolean, nullable
   - `position` : number

2. **AC2 — Type Task inféré** : `type Task = z.infer<typeof TaskSchema>` est exporté depuis `task.ts` — c'est la source de vérité unique pour le type Task.

3. **AC3 — FlowState schema** : `src/schemas/flow.ts` exporte un `FlowStateSchema` décrivant l'état éphémère du questionnaire cognitif (currentFlux, currentStep, answers, taskId en cours de tri).

4. **AC4 — AnalyticsSchema** : `src/schemas/analytics.ts` exporte un `AnalyticsSchema` décrivant les données persistées (micro-survey scores, timestamps, compteurs).

5. **AC5 — Persistence validation** : `src/lib/persistence.ts` exporte une fonction `validateAndLoad<T>(key: string, schema: ZodSchema<T>): T | null` qui :
   - Lit `localStorage.getItem(key)`
   - Parse le JSON
   - Valide avec `schema.safeParse()`
   - Si valide → retourne les données
   - Si invalide (JSON corrompu, schema mismatch, clé absente) → `localStorage.removeItem(key)` + `console.warn()` + retourne `null`

6. **AC6 — Tests schemas** : `src/schemas/task.test.ts` couvre :
   - Cas valide complet (tous les champs remplis)
   - Cas valide minimal (champs nullable à `null`)
   - Cas invalide : title vide, status inconnu, date mal formatée, id non-UUID, champ manquant
   - Edge cases : title avec espaces uniquement (si min 1), position négative ou zéro

7. **AC7 — Tests persistence** : `src/lib/persistence.test.ts` couvre :
   - localStorage valide → retourne données parsées
   - localStorage JSON invalide → reset + retourne `null`
   - localStorage schema mismatch → reset + retourne `null`
   - Clé absente → retourne `null` (pas de reset)

## Dev Notes

### Architecture Compliance

- **Schemas = source de vérité des types.** Tout type TypeScript lié aux données est inféré de Zod (`z.infer`), jamais défini manuellement.
- **`null` pour valeurs absentes** (jamais `undefined` en storage). Utiliser `.nullable()` sur les champs optionnels.
- **Dates ISO 8601 uniquement.** Utiliser `z.string().datetime()` pour la validation.
- **IDs UUID v4.** Utiliser `z.string().uuid()` pour la validation.
- **camelCase pour les clés JSON** : `createdAt`, `flowDurationMs`, `sourceFlux` (pas de snake_case).
- **Named exports uniquement** : `export function ...`, `export const ...`, `export type ...` — jamais de `export default`.

### Zod 4 — Points d'attention

- **Version : Zod 4.3.6** — déjà installé dans le projet (vérifier `package.json`).
- **Import** : `import { z } from 'zod'` (pas de `zod/v4` ou autre sous-chemin).
- **safeParse** retourne `{ success: true, data: T } | { success: false, error: ZodError }`.
- **z.enum()** accepte un array de strings : `z.enum(['inbox', 'backlog', 'active', 'archived'])`.
- **z.infer** : `type Task = z.infer<typeof TaskSchema>` — inférence automatique depuis le schema.

### Contexte métier pour les enums

**TaskStatus** :
- `inbox` : tâche vient d'être saisie dans le Vrac, pas encore triée
- `backlog` : triée et classée dans un quadrant en Réserve
- `active` : activée depuis la Réserve vers la Matrice Focus
- `archived` : complétée depuis la Matrice

**Quadrant** (Eisenhower) :
- `q1` : Critique — Important + Urgent
- `q2` : Essentiel — Important + Non urgent
- `q3` : Fausse urgence — Non important + Urgent
- `q4` : Optionnel — Non important + Non urgent

**SourceFlux** (questionnaire cognitif) :
- `flux1` : Projection des conséquences
- `flux2` : Ancrage sur objectifs
- `flux3` : Audit du stress
- `flux4` : Test du regret
- `manual` : classement manuel (sans questionnaire)
